# VoiceInput 语音对话功能 — 技术设计文档

## 一、功能目标

将 VoiceInput 从"语音转文字输入工具"升级为"语音对话助手"：

```
当前：  按住Fn说话 → 文字粘贴到光标处
升级后：按住Fn说话 → AI理解并回复 → 语音播放回复
        ↑                              │
        └── 随时按Fn打断，开始新一轮 ◄───┘

附加：所有对话内容自动记录到本地日志文件，可随时回溯查阅
```

## 二、整体架构

### 2.1 数据流全景

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户按住 Fn 说话                           │
└──────────────────────────┬──────────────────────────────────────┘
                           │ 音频流
                           ▼
              ┌────────────────────────┐
              │   STT (语音识别)         │  ← Apple SFSpeechRecognizer
              │   本地处理，免费，低延迟    │     已有模块: SpeechEngine
              └────────────┬───────────┘
                           │ 文字
                           ▼
              ┌────────────────────────┐
              │   LLM (大模型推理)       │  ← OpenAI Chat Completions API
              │   理解语义，生成回复       │     改造模块: LLMRefiner → ChatEngine
              │   system prompt 控制风格  │
              └────────────┬───────────┘
                           │ 回复文字
                           ▼
              ┌────────────────────────┐
              │   TTS (文字转语音)       │  ← OpenAI /v1/audio/speech
              │   自然语音合成            │     新增模块: TTSPlayer
              └────────────┬───────────┘
                           │ 音频数据
                           ▼
              ┌────────────────────────┐
              │   音频播放               │  ← AVAudioPlayer
              │   扬声器输出             │
              └────────────────────────┘
```

## 三、新增模块详细设计

### 3.1 ChatEngine — 多轮对话引擎

```
职责: 管理对话上下文，调用 OpenAI Chat Completions API

┌─────────────────────────────────────────────────┐
│                  ChatEngine                     │
├─────────────────────────────────────────────────┤
│                                                 │
│  属性:                                           │
│  ├── messages: [Message]    // 对话历史           │
│  ├── systemPrompt: String   // 人设 & 风格控制    │
│  ├── maxHistoryRounds: Int  // 最大保留轮数(默认10)│
│  └── currentTask: URLSessionDataTask?           │
│                                                 │
│  方法:                                           │
│  ├── send(text) → callback(replyText)           │
│  ├── clearHistory()                             │
│  └── cancel()                                   │
│                                                 │
└─────────────────────────────────────────────────┘

API 调用示意:

POST {apiBaseURL}/chat/completions
{
  "model": "gpt-4o-mini",
  "messages": [
    {"role": "system",  "content": "你是一个语音助手..."},  ← 风格控制
    {"role": "user",    "content": "上一轮用户说的话"},
    {"role": "assistant","content": "上一轮AI的回复"},
    {"role": "user",    "content": "这一轮用户说的话"}      ← 当前输入
  ],
  "temperature": 0.7,
  "max_tokens": 200          ← 限制回复长度，保持简短
}
```

System Prompt 设计（控制口语风格，不需要额外的压缩模型）:

```
你是一个友好的语音助手。你的回复会被转成语音播放给用户听。

规则:
- 回复简短，控制在1-3句话
- 用口语化的表达，像朋友聊天一样
- 不要用列表、编号、markdown格式
- 不要用书面语或学术用语
- 如果问题复杂，给出关键结论，不要展开细节
- 用中文回复，除非用户用英文提问
```

### 3.2 TTSPlayer — 语音合成与播放

```
职责: 调用 OpenAI TTS API，播放合成语音

┌─────────────────────────────────────────────────┐
│                  TTSPlayer                      │
├─────────────────────────────────────────────────┤
│                                                 │
│  属性:                                           │
│  ├── voice: String          // 语音角色          │
│  │   可选: alloy / echo / fable / onyx          │
│  │         nova / shimmer                       │
│  ├── speed: Double          // 语速 0.25~4.0    │
│  ├── audioPlayer: AVAudioPlayer?                │
│  └── currentTask: URLSessionDataTask?           │
│                                                 │
│  方法:                                           │
│  ├── speak(text) → callback(finished)           │
│  ├── stop()                                     │
│  └── previewVoice(voice) → 试听                  │
│                                                 │
└─────────────────────────────────────────────────┘

API 调用示意:

POST {apiBaseURL}/audio/speech
Headers:
  Authorization: Bearer sk-xxx
  Content-Type: application/json

Body:
{
  "model": "tts-1",           // tts-1(快) 或 tts-1-hd(高质量)
  "input": "要合成的文字",
  "voice": "nova",            // 语音角色
  "speed": 1.0,               // 语速
```

### 3.3 ConversationStore — 对话历史管理

```
职责: 维护多轮对话的上下文窗口

┌─────────────────────────────────────────────────┐
│              ConversationStore                   │
├─────────────────────────────────────────────────┤
│                                                 │
│  struct Message {                               │
│      role: "user" | "assistant"                 │
│      content: String                            │
│      timestamp: Date                            │
│  }                                              │
│                                                 │
│  属性:                                           │
│  ├── messages: [Message]                        │
│  └── maxRounds: Int = 10                        │
│                                                 │
│  方法:                                           │
│  ├── append(role, content)  // 添加一条消息       │
│  ├── buildAPIMessages()     // 构建API请求体      │
│  ├── clear()                // 清空历史          │
│  └── trimIfNeeded()         // 超出轮数时裁剪     │
│                                                 │
└─────────────────────────────────────────────────┘

上下文窗口策略:

  保留最近 N 轮对话 (默认10轮 = 20条消息)
  超出时从最早的开始丢弃
  system prompt 始终保留

  ┌─────────────────────────────────┐
  │ system: "你是语音助手..."        │ ← 永远保留
  │ user:   "第8轮问题"             │ ← 最早保留的
  │ asst:   "第8轮回复"             │
  │ user:   "第9轮问题"             │
  │ asst:   "第9轮回复"             │
  │ user:   "第10轮问题"            │ ← 当前输入
  └─────────────────────────────────┘
```

## 四、打断机制设计

### 4.1 打断场景分析

```
真实对话中，用户可能在 AI 回复的任何阶段打断:

场景1: AI正在"思考" (等待LLM返回)
┌──────────────────────────────────────────────────────┐
│  用户松开Fn → LLM请求中... → 用户按下Fn (打断!)       │
│                                                      │
│  处理: 取消LLM请求 → 立即开始新一轮录音                 │
└──────────────────────────────────────────────────────┘

场景2: AI正在"说话" (TTS播放中)
┌──────────────────────────────────────────────────────┐
│  TTS播放中 → 用户按下Fn (打断!)                        │
│                                                      │
│  处理: 停止播放 → 记录已播放部分 → 开始新一轮录音        │
└──────────────────────────────────────────────────────┘

场景3: AI正在"合成语音" (等待TTS返回)
┌──────────────────────────────────────────────────────┐
│  TTS请求中... → 用户按下Fn (打断!)                     │
│                                                      │
│  处理: 取消TTS请求 → 开始新一轮录音                     │
└──────────────────────────────────────────────────────┘
```

### 4.4 打断处理的核心逻辑 (伪代码)

```swift
// AppDelegate 中的 fnDown() 改造

func fnDown() {
    guard isEnabled else { return }

    switch chatState {
    case .idle:
        // 正常开始录音
        startNewRecording()

    case .recording:
        // 已经在录音，忽略
        break

    case .thinking:
        // 打断: 取消LLM请求
        chatEngine.cancel()
        logInterruption(phase: "thinking")
        startNewRecording()

    case .synthTTS:
        // 打断: 取消TTS请求
        ttsPlayer.cancel()
        logInterruption(phase: "tts_synthesis")
        startNewRecording()

    case .speaking:
        // 打断: 停止播放
        ttsPlayer.stop()
        logInterruption(phase: "speaking")
        startNewRecording()
    }
}

func logInterruption(phase: String) {
    chatLogger.logEvent(.interrupted(
        phase: phase,
        partialReply: currentReply,    // AI已生成的回复
        playedDuration: ttsPlayer.currentTime  // 已播放时长
    ))
    overlayPanel.updateText("⏸ 已打断")
}
```

### 4.5 被打断的回复如何进入上下文

```
问题: AI说到一半被打断了，对话历史里该记什么？

策略: 记录完整回复，但标注被打断

  ┌─────────────────────────────────────────────────┐
  │ messages 数组:                                   │
  │                                                 │
  │ user: "给我讲讲量子力学"                          │
  │ assistant: "量子力学是研究微观粒子行为的物理学      │
  │            分支，它的核心思想是..."                 │
  │            ↑ 完整回复都保留，AI知道自己说过什么      │
  │                                                 │
  │ user: "等等，先说说薛定谔的猫"  ← 打断后的新问题   │
  └─────────────────────────────────────────────────┘

  这样 AI 知道:
  ① 自己之前在讲量子力学
  ② 用户打断了（因为话题突然转了）
  ③ 可以自然地衔接新话题
```

## 五、对话日志系统

### 5.1 设计目标

```
┌─────────────────────────────────────────────────────────┐
│  需求:                                                   │
│  ① 所有对话内容自动记录，作为"通话记录/证据"               │
│  ② 每次对话会话(session)独立成一个日志文件                 │
│  ③ 记录完整信息: 时间、角色、内容、是否被打断等             │
│  ④ 人类可读格式，方便直接打开查看                          │
│  ⑤ 同时提供结构化格式(JSON)，方便程序解析                  │
└─────────────────────────────────────────────────────────┘
```

### 5.2 ChatLogger 模块设计

```
┌─────────────────────────────────────────────────┐
│                  ChatLogger                     │
├─────────────────────────────────────────────────┤
│                                                 │
│  属性:                                           │
│  ├── logDirectory: URL      // 日志存储目录       │
│  │   ~/Library/Logs/VoiceInput/Chats/           │
│  ├── currentSessionID: UUID // 当前会话ID        │
│  └── currentSessionStart: Date                  │
│                                                 │
│  方法:                                           │
│  ├── startSession()         // 开始新会话        │
│  ├── log(role, content)     // 记录一条消息       │
│  ├── logInterruption(phase, partial)  // 记录打断 │
│  ├── endSession()           // 结束会话          │

│                                                 │
└─────────────────────────────────────────────────┘
```

## 七、进阶优化 — 流式 LLM + 分段 TTS

```
目标: 不等LLM全部生成完再调TTS，而是边生成边合成，大幅降低感知延迟

标准流程 (串行):
  LLM生成全部文字 ──────────────► TTS合成全部音频 ──────► 播放
  [========= 2s =========]       [====== 1s ======]      [播放]
  总等待: ~3s

优化流程 (流式):
  LLM流式输出第1句 ──► TTS合成第1句 ──► 播放第1句
  LLM流式输出第2句 ──► TTS合成第2句 ──────────► 播放第2句 (无缝衔接)
  [=== 0.8s ===]      [= 0.5s =]     [播放...]
  总等待: ~1.3s (用户就听到声音了)

实现方式:
  ① LLM 请求加 "stream": true
  ② 按句号/问号/感叹号 分割文字
  ③ 每凑够一句就立即调 TTS
  ④ 音频队列顺序播放

  ┌─────────┐    句子1    ┌─────────┐   音频1   ┌──────────┐
  │  LLM    ├────────────►│  TTS    ├──────────►│  播放队列  │
  │  流式输出 ├────────────►│  并发请求 ├──────────►│  顺序播放  │
  │         ├────────────►│         ├──────────►│          │
  └─────────┘    句子3    └─────────┘   音频3   └──────────┘
```

这是进阶优化，建议第一版先用串行方式跑通，体验没问题后再加流式。
